# Design

DCP creation tool. Rust core with CLI and Tauri GUI.

## Layout

- `rust/crates/dcpwizard-core`: DCP assembly (CPL/PKL/ASSETMAP writers, MXF wrapping via postkit/asdcplib), encryption, KDM, export, job queue.
- `rust/crates/dcpwizard-cli`: clap CLI (create, encode, pipeline, kdm, certificate, verify, batch/daemon, serve, dashboard, ...).
- `gui/`: Tauri app; builds run through postkit::pipeline with the CLI as sidecar.
- Shares code via postkit (path dep at `extern/postkit`), asdcplib-rs (git), and dcpdoctor-core (git) for verify/info/report.
- CPL/PKL/ASSETMAP/VOLINDEX XML, `escape_xml`, and SRT parsing come from `postkit::packaging` / `postkit::subtitle_retime` (the app maps its configs onto those writers; no hand-rolled XML remains in cpl/pkl/assetmap).
- `~15` core modules are thin `pub use postkit::...` re-export shims.

## What is implemented and wired

- OV creation from video files or J2K dirs + WAV: demux, J2K encode (grok in-process FFI), real asdcplib MXF wrapping, CPL/PKL/ASSETMAP/VOLINDEX generation, SHA-1 hashes. `create --container` selects 2K/4K scope, flat, or full CPL dimensions; `create --container-dims WxH` sets arbitrary dimensions (validated positive, even, and within the 2K 2048x1080 / 4K 4096x2160 bound). The dims thread into the CPL ScreenAspectRatio and the MainPicture stored/active area.
- Background/pad colour (`create --pad-color <RRGGBB>`, dom#1042): the head/tail pad frames are filled with a caller-given sRGB colour instead of black. The colour runs through postkit's real Rec.709 RGB -> DCI X'Y'Z' DCDM transform (`colour::rgb_to_xyz_inplace`) before J2K encoding, so the stored codestream carries correct X'Y'Z' code values (verified: a decoded pad frame's XYZ matches an independent Rec.709->XYZ+2.6-gamma computation within 1 code value). Note: grok's `apply_xyz_transform` cparameter works (an earlier inert reading came from a stale cached grokj2k-sys build; run `cargo clean` if colour output looks wrong). `create --video` sets it true for SDR sources and for the generic tonemap branch, and false on the HDR-LUT branch (the LUT already outputs X'Y'Z'). Pad frames use postkit `colour::rgb_to_xyz_inplace` with the flag off. grok's transform and postkit's differ slightly (red: [2817,2183,870] vs [2914,2258,898], different linearization), both valid X'Y'Z'; harmonizing them is an open todo.
- Reel splitting (`create --reel-length <minutes>`, reel.rs): splits the encoded frame sequence, audio and subtitles into per-reel ranges and wraps one picture/sound/subtitle MXF set per reel, then emits a multi-reel CPL over a shared PKL/ASSETMAP. Boundaries are in whole frames at the edit rate; audio is sliced sample-accurately (frames * sample_rate/fps) and padded to the reel frame count; DCST cues are re-split by reel and rebased to reel-local time. Every reel is at least 1s (a sub-second remainder merges back). postkit's wrapper consumes every input file, so picture is wrapped from an explicit per-reel file list and audio/subtitle from temp files. The single-reel path (flag absent) is unchanged.
- Explicit reel splits (dom#2964): `create --split-at <tc>[,<tc>...]` splits at HH:MM:SS or HH:MM:SS:FF timecodes (converted to frame boundaries); `create --split-chapters` (and `pipeline --split-chapters`) read the source's chapter marks via `ffprobe -show_chapters`. Both feed explicit frame boundaries into the same multi-reel path as `--reel-length` (audio/subtitle split identically), and reject any boundary that would make a reel shorter than 1s. `--split-at` and `--reel-length` are mutually exclusive.
- Input decode range (`create`/`pipeline --input-range full|legal`): forces the source's decode range instead of trusting its metadata, correcting wrong or absent range flags. postkit's raw-RGB ffmpeg decode is frozen, so the correction re-decodes the source into a lossless FFV1/gbrp16le intermediate with `scale=in_range=<full|tv>:out_range=full` and the encode reads that. Video input only.
- Disk writer (`format-drive`, `check-drive`, disk.rs, dom#2095/dom#2112): `format-drive <target> --fs ext2|ext3 [--label] --yes [--image]` shells out to mkfs.ext2/ext3 (`-L` label). Safety: it refuses any target that (or whose partitions) appear in /proc/mounts, requires `--yes`, and without `--image` requires a block device (`--image` allows a regular file for tests/loopback). `check-drive <target>` reports the filesystem type and label without modifying, via `blkid -o export` when present, else by parsing the ext superblock directly (magic 0xEF53 at 0x438, journal/feature flags to tell ext2/3/4 apart, label at 0x478) so it works unprivileged on image files.
- HFR frame rates (hfr.rs): legal SMPTE rates 24/25/30/48/50/60/96/100/120 (Interop 24/25/30/48); 4K is capped at 30 fps per the DCI HFR addendum. `create_dcp` and the CLI/GUI pipelines reject an illegal fps/resolution combo loudly before encoding; the edit rate threads through both CPL and MXF.
- Supplemental Version File (VF) creation (`create-vf`): references the OV's real asset ids for unchanged reels, wraps replacement essence (or copies an already-wrapped MXF) and registers it under its real embedded asset id in CPL/PKL/ASSETMAP, carries an `<OriginalPackagingList>` marker so dcpdoctor detects a supplemental. Validating the VF with `--ov` resolves cross-refs; alone it yields the supplemental-OV-not-provided warning, not a hard cross-ref error. `--replace-picture`/`--replace-sound` swap essence tracks; `--replace-subtitle`/`--add-subtitle REEL=PATH` (with `--subtitle-language`) wrap an SRT (converted to DCST) or a supplied SMPTE XML (unchanged) into a per-reel timed-text MXF and set the reel's MainSubtitle. A subtitle-only VF references the OV picture/sound by id and ships only the new subtitle MXF, so it can be authored from the OV's CPL/PKL/ASSETMAP without the OV essence (dom#1062).
- Assemble OV from existing DCPs (`assemble --input <dcp>... --output --title`, assemble.rs, dom#1675): builds one new composition (single CPL) whose reels are the inputs' reels in order (e.g. short + feature). Every referenced MXF is copied byte-identical and referenced by its existing asset UUID; a fresh CPL/PKL/ASSETMAP/VOLINDEX is written. Inputs must agree on standard, edit rate and picture resolution, else it fails loud listing the mismatch. Encrypted inputs are refused because a new CPL id would break the KDM's CPL binding.
- Edit DCP metadata (`edit --input [--output] [--title] [--annotation] [--content-kind] [--issuer]`, edit.rs, dom#1127): rewrites the single CPL's title / annotation / content kind / issuer in place (or into `--output`, copied first), assigns a new composition id, and refreshes the CPL's PKL and ASSETMAP entries (new id, new hash/size). Essence MXFs and their asset ids are untouched. Each file is written atomically (temp + rename). Encrypted DCPs are refused (the KDM binds the CPL id). Metadata fields only; the digest's reel reorder/length edits are out of scope.
- Multi-composition package (`create-multi --compositions <manifest>`, multi_cpl.rs `create_multi_composition`): one CPL per manifest entry, each with its OWN picture/audio/subtitle, over one shared PKL/ASSETMAP. The manifest is a JSON array of `{title, j2k_dir, audio?, subtitle?, subtitle_language?, kind?}`. Single reel per composition. Contrast `create --versions`, which writes several CPLs over one SHARED essence set. Reuses the same wrap primitives (mxf_wrap, the versions.rs sound/subtitle helpers); with `--encrypt` it writes one key bundle per CPL.
- Encryption: content keys generated with the OS CSPRNG, essence AES-128 encrypted at wrap time (`postkit::mxf_wrap`, HMAC per SMPTE), PKL/ASSETMAP hashes taken from the final encrypted files, KeyIds written into the CPL. `create --encrypt` requires `--key-out <path>` (clap-enforced); the plaintext keys are written only there, never next to the DCP. The GUI mirrors this: encrypting requires a Key Output File.
- SMPTE (and Interop) KDM generation, batch, DKDM rewrap (real signed ETM). `kdm --keys <key-out file>` binds the KDM to the DCP's actual image (MDIK) and audio (MDAK) content keys. `kdm --format smpte|interop` (also on `kdm-batch`) selects the ETM: Interop uses postkit's digicine 134-byte block, legacy output that needs real-gear validation. `kdm-batch` takes repeated `--cert` or a `--cert-dir` of cinema certs and writes one signed KDM per recipient. Certificate chain generation and inspection.
- Decrypt (`decrypt --input --output`, decrypt.rs): turns an encrypted DCP into a cleartext DCP of the same reel structure. Content keys come from a KDM plus the recipient RSA private key (`--kdm --recipient-key`, unwrapped via postkit `unwrap_kdm`) or the DCP's KEYS.json (`--keys`); one of those two sources is required. Each encrypted essence MXF is read frame by frame through asdcplib's AES decrypt + HMAC context keyed by the MXF's own cryptographic_key_id, so a wrong key or a KDM that does not cover a KeyId fails loud (the OAEP unwrap, or the per-frame MIC, rejects it); non-encrypted assets copy byte-identical. The output CPL drops all KeyId/encryption elements, so it gets a new CPL id with fresh PKL/ASSETMAP/VOLINDEX hashes. Rewrapped essence gets a new asset UUID (postkit's wrapper always mints one, there is no override), so the decrypted package is fresh, not a byte-identical structural clone. Key material stays in memory only, never in logs, errors or temp files. Encrypted picture and sound decrypt (what `create --encrypt` produces); an encrypted timed-text/aux track fails loud rather than shipping untested handling.
- Encrypted-source transcode (`transcode-dcp --kdm --recipient-key | --keys`): the same key sources let transcode-dcp decrypt each source picture frame in memory before grk_decompress, then re-encode and rewrap cleartext exactly as the plaintext path; sound/subtitle go through the decrypt path so the cleartext output stays coherent. Without key material an encrypted source is refused loud. The J2K temp frames written during transcode are plaintext by design.
- KDM distribution (cinema.rs, kdm_log.rs, kdm_template.rs, flm.rs, email.rs, cert_fetch.rs; shared helpers in store.rs). State files live under the XDG data dir (`~/.local/share/dcpwizard/`), each overridable per command, all written atomically (temp + rename):
  - Cinema/screen database (`cinemas.json`, `--db`). A cinema has contact emails and notes; each screen holds a recipient certificate as a file path or inline PEM, with the cert serial, SHA-1 thumbprint and subject cached for search. No private key material is stored. `cinema add|list|remove|add-screen|remove-screen|search|import-flm`; search matches cinema/screen name or cert serial/thumbprint substring. `kdm-batch --cinema <name>`/`--screen <cinema>/<screen>` resolves recipients from the db (inline certs are materialized to temp files); `--cert`/`--cert-dir` still work.
  - History log (`kdm-history.jsonl`, `--history-file`): every successful `kdm`/`kdm-batch` appends one JSON line of metadata only (UTC timestamp, CPL id, title, recipient subject + serial, validity window, output path, format) — never key material or KDM bodies. `kdm-history` lists and filters by title, recipient or date range.
  - Validity templates (`kdm-templates.json`): a named relative window = optional start offset from now + duration, emitted in a chosen UTC offset. `kdm-template add|list|remove`; `kdm`/`kdm-batch --template <name>` expands the window, and an explicit `--valid-from`/`--valid-to` overrides it. Expansion produces absolute ISO timestamps that flow through the same window resolution as a directly supplied window.
  - FLM-x import (`cinema import-flm`): parses an ISDCF FLM-x / SMPTE ST 430-7 FacilityListMessage (roxmltree, matched on element local names so the schema revision need not be pinned), taking the facility name, contact emails and one screen per device with its recipient (leaf, non-CA) X.509 certificate. A non-FacilityListMessage root fails loud.
  - Email delivery (`kdm`/`kdm-batch --email-to --smtp-config`, lettre 0.11 over rustls): the SMTP config is a TOML file (host, port, security tls/starttls/none, username, password, from, optional subject/body templates with `{title}`/`{cinema}`); the password is redacted in Debug and never logged. `kdm-batch --cinema` sends one email per cinema, its KDMs zipped, to the cinema's contact emails plus any `--email-to`; `--email-only-additional` sends only to `--email-to`.
  - `cert-fetch --vendor --serial [--type] -o` downloads a recipient certificate and validates it as parseable X.509 before writing. Supported endpoints are the credential-free public FTP servers: dolby/doremi (`ftp://ftp.cinema.dolby.com/Certificates/<first3>xxx/`, the `*-<serial>.cert.sha256.pem` leaf) and qube (`ftp://certificates.qubecinema.com/SMPTE-<type>/`). Other vendors error, telling the user to obtain the cert from the vendor. FTP transfer shells out to the system `curl` binary; the serial is validated to alphanumerics and passed as an argument, never through a shell.
- Stereoscopic 3D (`create --right-eye`): both eyes encoded at the same settings, wrapped into one ST 429-10 picture MXF (postkit `wrap_stereoscopic`), CPL carries the `MainStereoscopicPicture` element (429-10/2008 namespace, EditRate = composition rate, FrameRate doubled). Validated against the clairmeta 429-10 XSD.
- MCA channel labeling: 5.1 input is expanded to 16 PCM channels, with L/R/C/LFE/Ls/Rs in the first six positions and ten silent channels. `create --audio-input-order lrc-ls-rs-lfe` explicitly remaps that common source order; it never guesses. ST 429-12 labels cover the canonical 5.1 channels, and `create --hi-channel/--vi-channel` add standalone HI/VI-N labels.
- Dolby Atmos (`create --atmos`): wraps a Dolby Atmos / DCData bitstream (`EssenceType::Atmos`) and registers a ST 429-18 `AuxData` element (Dolby `2012/AD` namespace, IAB UL) in CPL/PKL/ASSETMAP. Structure validates; real conformance needs real Atmos essence.
- Sign-language video (`create --sign-language-video <file> --sign-language-lang <rfc5646>`, sign_language.rs, dom#1602, ISDCF Doc 13): the input is encoded to VP9 (480x640, 24 fps, ~576 kbps, yuv420p, 2s chunks via ffmpeg webm_chunk) and packed into 288,000-byte PCM blocks (20-byte big-endian header MARKER/Lv/Lb/Le/MARKER + webm header + VP9 segment + null pad) carried on channel 15 of the 16-channel MainSound track. asdcplib's MCA parser labels channel 15 SLVS (UL 06.0e.2b.34.04.01.01.0d.0d.0f.03.02.01.01.00.00, per-channel language `SLVS:<lang>`, needs a trailing channel-16 token so the language applies), and the CPL CompositionMetadata carries the `http://isdcf.com/2017/10/SignLanguageVideo` ExtensionMetadata with the Language Tag. Packing verified against the ISDCF reference encoder; channel/UL/CPL against libdcp. The MCA read-back exposes only counts (main + 1 SLVS channel), not per-channel ULs.
- verify/info/report via dcpdoctor-core; SRT to subtitle-XML conversion; supplied SMPTE subtitle XML is wrapped unchanged, preserving authored placement and styling. Subtitle/watermark burn-in is separate.
- Subtitle input formats + authoring (subtitle.rs, W6): `create --subtitle` accepts `.srt`, `.ass`/`.ssa` (dom#1462), `.pac` (dom#1719, Latin codepage), `.mks`/`.mkv` (dom#3131, via ffmpeg), `.fcpxml` (dom#2909) and an Interop `DCSubtitle` XML with PNG bitmap subs (dom#1376); a supplied SMPTE DCST XML is still wrapped unchanged. postkit's parsers produce `StyledCue`s; the styled DCST renderer carries per-run italic/bold/underline/colour as inline `<dcst:Font>` spans and per-cue alignment/position. Placement controls (dom#1729/#2796): `--subtitle-halign/--subtitle-valign/--subtitle-vposition` (top-anchored cues grow downward). 3D depth (dom#1359): `--subtitle-zposition` emits the ST 428-7 Zposition on every cue (the timed-text descriptor's Z-Position-In-Use flag is not set by the asdcplib writer). RTL (dom#860): `--subtitle-rtl auto|on|off` reshapes+reorders Hebrew/Arabic to visual order via postkit bidi. Wrap (dom#1626): `--subtitle-wrap <chars>` wraps overlong lines on whitespace via postkit wrap. Font (dom#1023): `--subtitle-font <ttf/otf>` subsets to the used glyphs (postkit font_subset, SMPTE 640 KB limit, fails loud over it; `--subtitle-no-subset` opts out) and embeds it in the timed-text MXF referenced by a `LoadFont` whose urn matches the embedded ancillary-resource id (postkit `MxfWrapOptions.resource_ids`). Bitmap PNGs embed the same way with `<dcst:Image>` refs. Cross-reel font dedup (dom#2533): under reel splitting the font is embedded per reel MXF (SMPTE embeds fonts inside each timed-text MXF) but referenced by one shared asset id in every reel's XML. Verified end-to-end: ass/font/interop-png creates pass dcpdoctor with 0 errors; the styled DCST validates against the ST 428-7 XSD.
- Standalone subtitle editing (`subtitle-edit`, subtitle_edit.rs, dom#828/#2071): `--list` cues, `--shift-ms` all cues, or `--index N` with `--text`/`--set-start-ms`+`--set-end-ms`, written back as SRT. Loads any parsable format; edits source files only, never subtitles inside a finished DCP.
- HDR source video requires `create --hdr-to-dci-lut <lut>`. `--allow-generic-hdr-tonemap` enables FFmpeg tone mapping with a warning, for explicitly accepted non-delivery use.
- Transcode/import via ffmpeg, frame extraction, dv-inject via dovi_tool, loudness via loudnorm.
- Create-time audio processing (audio_route.rs, loudness.rs, main.rs `prepare_create_audio`, applied to explicit `--audio` and to demuxed audio in both the video and J2K-dir branches): filename channel auto-routing (dom#2134) when `--audio` is a directory of mono WAVs (`name_L.wav`/`_R`/`_C`/`_Lfe`/`_Ls`/`_Rs`/... routed to canonical DCP lanes, missing lanes silent, non-mono or unknown suffix fails loud); stereo->5.1 upmix (`create --upmix a|b`, dom#921/#1080, postkit `upmix_wav`); loudness normalization (`create --loudness-target leqm=<db>|lufs=<v>` with `--true-peak-ceiling`, dom#1382, postkit `adjust_loudness`, refused loud if the gain would breach the true-peak ceiling). Order is route -> upmix -> loudness, before pull-up and sign-language packing. Standalone `crossfade --a --b -o --overlap` (dom#374, equal-power join, postkit `crossfade_join_wav`) and `mid-side-decode -i -o --mid --side` (dom#3020, postkit `decode_mid_side_wav`) subcommands.
- Encode QoL on `create` (encode_qol.rs): `--start-at HH:MM|RFC3339|+offset` blocks until a wall-clock time before encoding (dom#2359); `--resume` reuses the on-disk J2K frame prefix via postkit `encode_video_pipeline_resumable`, guarded by a persisted `.dcpwizard-encode.json` that must match source+params (dom#344, survives restarts); progress prints a running average-fps ETA (dom#502); `--shutdown-when-done` powers off after a clean encode, resolved loud up front so a missing shutdown binary fails before the long run (dom#1394; never runs in tests).
- SQLite version tracker and dashboard (including serve), daemon/batch IPC, shell completion.
- GUI: shortcuts, recents, filter, preview, timeline view, sidecar setup; KDM panel collects signer cert/key + keys file; encode honours the selected frame rate; OS file drop uses the Tauri webview drag-drop event.

## Not implemented (de-advertised)

These have no working path and are not advertised: DTS:X (blocked on a confirmed DataEssenceCoding UL, see DESIGN_TODO; the `MxfType::DtsX` variant was removed so it fails loud rather than emitting an Atmos UL). Atmos KDM (essence-encrypted aux track in a KDM) is not implemented. `cli_flags_test.sh` invokes each GUI command line and fails on clap parse errors.

Subtitle packaging into a DCP timed-text track is real: `create --subtitle` converts SRT/ASS/PAC/MKS/FCPXML/Interop-PNG to ST 428-7 DCST XML, and supplied SMPTE XML is wrapped unchanged. Both paths register the timed-text MXF in the reel/CPL/PKL/ASSETMAP. Placement, RTL, wrap, font embedding and 3D Zposition are wired (see the subtitle bullet above).

Deleted (dead, zero callers): dcp_diff, plugin, preferences, geometry, and the prores re-export shim.
